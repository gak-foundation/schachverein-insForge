# Better Auth System Implementierung

## Zusammenfassung
Vervollständigung des Better Auth Systems für den Schachverein mit Rate Limiting, Account Lockout, verbesserter RBAC-Integration und Audit Logging.

## Analyse des aktuellen Zustands

### Bereits implementiert
- Better Auth Basis-Konfiguration (`better-auth.ts`)
- Client-Side Hooks (`client.ts`)
- Session Management (`session.ts`)
- RBAC Permission System (`permissions.ts`)
- Security Headers (`security-headers.ts`)
- Middleware/Routenschutz (`proxy.ts`)
- Auth API Routes (`[...all]/route.ts`)
- Login/Signup Seiten
- Passwort-Reset und Email-Verifikation Actions

### Fehlende Features
- Rate Limiting (keine Implementierung vorhanden)
- Account Lockout nach fehlgeschlagenen Versuchen
- Audit Logging Integration in Actions
- Verbesserte Member-User Verknüpfung
- Sichere Token-Generierung für Einladungen

## Implementierungsplan

### Phase 1: Rate Limiting
**Ziel**: Schutz vor Brute-Force-Angriffen

1. **Neue Datei**: `src/lib/auth/rate-limit.ts`
   - Redis-basiertes Rate Limiting
   - Konfiguration für Login, Register, Password Reset
   - Per-User und Per-IP Limits

2. **Integration**: `src/lib/auth/better-auth.ts`
   - Rate Limiting in Better Auth Hooks aktivieren
   - Falls Redis nicht verfügbar: In-Memory Fallback für Development

3. **Environment**: `.env.example` erweitern
   - `RATE_LIMIT_LOGIN_MAX=5`
   - `RATE_LIMIT_WINDOW_MS=900000` (15 Minuten)

### Phase 2: Account Lockout
**Ziel**: Automatische Sperrung nach wiederholten Fehlversuchen

1. **Neue Datei**: `src/lib/auth/account-lockout.ts`
   - Tracking fehlgeschlagener Login-Versuche
   - Automatische Sperrung nach 5 Fehlversuchen
   - 30-Minuten-Sperrzeit
   - Reset nach erfolgreichem Login

2. **Datenbank**: Schema bereits vorhanden (`failedLoginAttempts`, `lockedUntil` in `authUsers`)

3. **Integration**: Hooks in `better-auth.ts`
   - `beforeLogin` Hook: Prüfung auf Lockout
   - `afterLogin` Hook: Reset bei Erfolg
   - `onError` Hook: Inkrementieren bei Fehlschlag

### Phase 3: Audit Logging
**Ziel**: Nachvollziehbarkeit sicherheitsrelevanter Aktionen

1. **Neue Datei**: `src/lib/audit.ts` (existiert bereits, aber erweitern)
   - `logAudit()` Funktion erweitern
   - Kategorien: AUTH, MEMBER, FINANCE, ADMIN
   - Automatische IP- und User-Agent-Erfassung

2. **Integration**: In alle Server Actions
   - Login/Logout
   - Mitgliederverwaltung
   - Rollenänderungen
   - Passwort-Änderungen

### Phase 4: Member-User Verknüpfung
**Ziel**: Nahtlose Verbindung zwischen `members` und `authUsers`

1. **Erweiterung**: `src/lib/actions.ts`
   - `linkMemberToUser()`: Verknüpfung herstellen
   - `inviteMember()`: Erweitern mit Token-Generierung
   - Automatische Role-Synchronisation

2. **Neue Datei**: `src/lib/auth/invitations.ts`
   - Sichere Einladungs-Token
   - Zeitlich begrenzte Gültigkeit (7 Tage)
   - Einmalige Verwendung

### Phase 5: Sicherheitsverbesserungen
**Ziel**: Defense in Depth

1. **Erweiterung**: `src/lib/auth/better-auth.ts`
   - Cookie-Cache aktivieren (für bessere Performance)
   - Trusted Origins erweitern
   - Session-Fingerprinting (optional)

2. **Erweiterung**: `src/proxy.ts`
   - Rate Limit Checks für sensible Routes
   - CSRF-Token Validierung

3. **Erweiterung**: `.env.example`
   - Sicherheitsrelevante Defaults dokumentieren

## Sicherheitsüberlegungen

### Rate Limiting Strategie
- Login: 5 Versuche pro 15 Minuten pro IP + Email
- Passwort-Reset: 3 Versuche pro Stunde
- Registrierung: 3 Versuche pro Stunde pro IP

### Account Lockout Strategie
- 5 fehlgeschlagene Versuche = 30 Minuten Sperre
- Automatischer Reset nach erfolgreichem Login
- Manuelles Entsperren durch Admin möglich

### Audit Logging
- Alle Auth-Aktionen loggen
- Sensible Daten niemals loggen (Passwörter, Tokens)
- DSGVO-konforme Speicherung (IP-Anonymisierung optional)

## Offene Fragen

3. **Session-Lifetime**: Aktuell 7 Tage - ist dies angemessen für einen Schachverein?
