# Multi-Tenancy SaaS-Implementierung für Schachverein

## Ziel
Transformation des Single-Tenant Schachverein-Systems in ein echtes Multi-Tenant SaaS-Produkt (wie schach.studio), bei dem mehrere unabhängige Vereine auf derselben Plattform isoliert voneinander arbeiten können.

## Analyse des aktuellen Stands

### Stärken (bereits SaaS-ready)
- **Technologie-Stack**: Next.js 16, Drizzle ORM, Better Auth, BullMQ - alles Industriestandards
- **RBAC-System**: 23 Berechtigungen mit 8 Rollen bereits implementiert
- **Sicherheit**: Rate Limiting, Account Lockout, Security Headers, Audit Logging
- **Hintergrund-Jobs**: BullMQ/Redis für skalierbare Job-Verarbeitung

### Aktuelle Einschränkung
Das Schema in `src/lib/db/schema.ts` hat keine `club_id` Spalte - alle Daten gehören implizit zu einem einzigen Verein.

## Implementierungsplan

### Phase 1: Datenbank-Schema-Erweiterung (Kritisch)

#### 1.1 Clubs-Tabelle erstellen
Neue Tabelle `clubs` mit:
- `id`: UUID Primary Key
- `name`: Vereinsname
- `slug`: Eindeutiger Subdomain-Identifier (z.B. "sc-springer")
- `logoUrl`: Optionales Vereinslogo
- `website`: Optionale Vereinswebsite
- `address`: JSON für Adresse (Straße, PLZ, Ort)
- `contactEmail`: Kontakt-E-Mail
- `plan`: Abo-Plan (free, pro, enterprise)
- `stripeCustomerId`: Stripe Kunden-ID
- `stripeSubscriptionId`: Stripe Abo-ID
- `subscriptionStatus`: active, past_due, canceled, etc.
- `subscriptionExpiresAt`: Ablaufdatum
- `features`: JSON für aktivierte Features pro Plan
- `settings`: JSON für Vereins-spezifische Einstellungen
- `isActive`: Boolean für Account-Status
- `createdAt`, `updatedAt`: Timestamps

#### 1.2 Club-Members Verknüpfungstabelle
Da Mitglieder theoretisch mehreren Vereinen angehören können (z.B. Trainer), wird eine `clubMemberships` Tabelle benötigt:
- `id`: UUID Primary Key
- `clubId`: Foreign Key zu clubs
- `memberId`: Foreign Key zu members
- `role`: Rolle im spezifischen Verein (kann vom globalen Role abweichen)
- `joinedAt`: Beitrittsdatum
- `isPrimaryClub`: Boolean für Hauptverein
- `status`: active, inactive

#### 1.3 Bestehende Tabellen erweitern
Allen bestehenden Tabellen `clubId` hinzufügen:
- `members` → über clubMemberships (siehe oben)
- `teams` → `clubId` notNull
- `seasons` → `clubId` notNull
- `tournaments` → `clubId` notNull
- `events` → `clubId` notNull
- `payments` → `clubId` notNull
- `documents` → `clubId` notNull
- `contributionRates` → `clubId` notNull
- `newsletters` → `clubId` notNull
- `auditLog` → `clubId` nullable (für Club-Admins)

#### 1.4 Auth-User Erweiterung
Tabelle `authUsers` erweitern um:
- `activeClubId`: UUID (derzeit ausgewählter Verein)
- `ownedClubs`: Array von club IDs (für Gründer)

### Phase 2: Session-Management & Context

#### 2.1 Session-Erweiterung
- `src/lib/auth/session.ts` erweitern um `clubId`
- Session enthält immer: `user.id`, `user.role`, `user.activeClubId`
- Falls `activeClubId` null → Nutzer muss Verein wählen oder erstellen

#### 2.2 Club-Context Provider
- React Context für aktiven Verein
- Server Component Wrapper für Club-Validation
- "Club Switcher" UI-Komponente (ähnlich wie bei Vercel/Linear)

#### 2.3 Subdomain-basierte Club-Detection
- Middleware erweitern: `sc-springer.localhost:3000` → automatisch Club "sc-springer"
- Custom Domain Support vorbereiten

### Phase 3: Daten-Isolation implementieren

#### 3.1 Query-Filter-Funktion
Neue Utility `src/lib/db/club-filter.ts`:
```typescript
export function withClubFilter<T extends { clubId: string }>(
  query: any,
  clubId: string
) {
  return query.where(eq(query.clubId, clubId));
}
```

#### 3.2 Server Actions anpassen
Alle Actions in `src/lib/actions.ts` erweitern:
- Session validieren
- `clubId` aus Session extrahieren
- Automatisch `clubId` auf alle Queries anwenden
- Bei CREATE-Operationen: `clubId` setzen

#### 3.3 Audit-Log Erweiterung
- Audit-Log muss `clubId` enthalten
- Club-Admins sehen nur Logs ihres Vereins
- Super-Admins (wir) sehen alle Logs

### Phase 4: Club-Onboarding-Flow

#### 4.1 Vereinserstellung
- Wizard: Vereinsname → Slug prüfen → Kontaktdaten → Plan wählen
- Automatische Slug-Generierung mit Prüfung
- Initiales Setup: Erster Benutzer wird Admin

#### 4.2 Einladungssystem
- Bestehende Mitglieder zu Verein einladen
- Email-Invitation mit Token
- Bulk-Import von Mitgliedern aus CSV

#### 4.3 Plan-Auswahl
- Free: Bis 30 Mitglieder, Grundfunktionen
- Pro: Unbegrenzte Mitglieder, Turniere, API-Zugriff
- Enterprise: Custom Features, Dedicated Support

### Phase 5: Stripe-Integration

#### 5.1 Stripe-Konfiguration
- Stripe Account Setup
- Products/Prices: Free (€0), Pro (€29/Monat), Enterprise (Custom)
- Checkout Sessions für Pro-Upgrade

#### 5.2 Webhook-Handler
- `stripe.webhooks` Route für Events:
  - `checkout.session.completed` → Subscription aktivieren
  - `invoice.paid` → Status aktualisieren
  - `invoice.payment_failed` → Grace Period starten
  - `customer.subscription.deleted` → Plan auf Free downgraden

#### 5.3 Feature-Gates
- `src/lib/billing/features.ts` mit Feature-Matrix pro Plan
- Server-seitige Prüfung vor Feature-Nutzung
- UI zeigt Upgrade-Prompt bei gesperrten Features

### Phase 6: Super-Admin Dashboard

#### 6.1 Admin-Bereich
- Übersicht aller Vereine
- Club-Stats: Mitgliederzahl, Aktivität, Plan-Status
- Billing-Übersicht
- Support-Ticket-System (vorbereiten)

#### 6.2 Impersonation
- Als Club-Admin einloggen für Support
- Zeitlich begrenzte Impersonation-Token

### Phase 7: Migration & Testing

#### 7.1 Datenbank-Migration
1. `npm run db:generate` für neue Migration
2. Bestehende Daten: Einen "Default-Club" erstellen
3. Alle bestehenden Daten diesem Club zuweisen
4. `clubId` auf NOT NULL setzen (nach Migration)

#### 7.2 Testing-Strategie
- Unit Tests für Club-Isolation
- Integration Tests: User A darf User B's Daten nicht sehen
- Stripe Webhook Tests
- Performance Tests mit Multi-Tenant Queries

### Phase 8: Deployment & Monitoring

#### 8.1 Infrastructure
- PostgreSQL Schema für Multi-Tenancy optimiert
- Redis für Sessions (für Skalierung)
- Separate Stripe Live/Test Keys

#### 8.2 Monitoring
- Metriken pro Club: API-Calls, Storage, Active Users
- Alerts bei Subscription-Problemen
- Error Tracking mit Club-Context

## Abhängigkeiten & Reihenfolge

1. **Phase 1** muss zuerst abgeschlossen sein (Schema-Änderungen)
2. **Phase 2** kann parallel zu Phase 3 beginnen
3. **Phase 3** (Isolation) ist kritischer Pfad für alle anderen Phasen
4. **Phase 4** (Onboarding) kann entkoppelt entwickelt werden
5. **Phase 5** (Stripe) benötigt Phase 3
6. **Phase 6** (Admin) kann entkoppelt entwickelt werden

## Offene Fragen an den Nutzer

1. **Plan-Preise**: Sind €29/Monat für Pro und Custom für Enterprise akzeptabel?
2. **Free-Tier Limits**: 30 Mitglieder als Grenze angemessen?
3. **Migration**: Gibt es bereits produktive Daten, die migriert werden müssen?
4. **Subdomain vs. Path**: Bevorzugt ihr `sc-springer.schach.studio` oder `schach.studio/sc-springer`?
5. **Custom Domains**: Soll später Custom Domain Support (sc-springer.de → unsere App) möglich sein?
6. **Stripe**: Besteht bereits ein Stripe Account, oder soll ich Test-Integration verwenden?

## Risiken & Mitigationen

| Risiko | Mitigation |
|--------|------------|
| Daten-Leak zwischen Clubs | Strikte Tests, Code Review für alle Queries |
| Performance bei vielen Clubs | Indizes auf clubId, Query-Optimierung |
| Stripe Webhook-Fehler | Idempotency, Retry-Logik, Monitoring |
| Schema-Änderungen sind destruktiv | Backup, Migration-Tests, Rollback-Plan |
