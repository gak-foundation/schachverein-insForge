# Better Auth Migration Plan

## Overview
Migration from NextAuth v5 to Better Auth for the Schachverein chess club management application.

## Prerequisites
- [ ] Backup database before migration
- [ ] Review Better Auth documentation: https://www.better-auth.com/
- [ ] Understand the Drizzle adapter for Better Auth

## Phase 1: Dependencies & Installation

### 1.1 Remove NextAuth dependencies
```bash
npm uninstall next-auth @auth/drizzle-adapter
```

### 1.2 Install Better Auth
```bash
npm install better-auth
npm install --save-dev @better-auth/drizzle-adapter
```

## Phase 2: Database Schema Migration

### 2.1 Update `src/lib/db/schema.ts`

Better Auth uses its own table structure. Replace NextAuth tables with Better Auth schema:

**Remove/Replace these tables:**
- `users` - Keep but rename to `auth_users` or use Better Auth's structure
- `accounts` - Remove (Better Auth handles this)
- `sessions` - Remove (Better Auth handles this)
- `verificationTokens` - Remove (Better Auth handles this)
- `refreshTokens` - Can be kept or removed

**Add Better Auth tables:**
- `user` (Better Auth's user table)
- `session` (Better Auth's session table)
- `account` (Better Auth's account table)
- `verification` (Better Auth's verification table)

**Key decisions needed:**
- How to map `member_role` enum to Better Auth's role system?
- How to migrate existing user data?
- Keep the `members` ↔ `users` relationship?

### 2.2 Migration Strategy

**Option A: Fresh start (Recommended)**
- Create new Better Auth tables
- Run data migration script to transfer users
- Keep old tables as backup, then drop later

**Option B: In-place migration**
- Modify existing tables to match Better Auth schema
- Riskier, requires careful column mapping

**Decision:** Go with Option A - safer for production.

## Phase 3: Better Auth Configuration

### 3.1 Create new auth config: `src/lib/auth/better-auth.ts`

```typescript
import { betterAuth } from "better-auth"
import { drizzleAdapter } from "@better-auth/drizzle-adapter"
import { db } from "@/lib/db"

export const auth = betterAuth({
  database: drizzleAdapter(db, {
    provider: "pg", // PostgreSQL

  emailAndPassword: {
    enabled: true,

  socialProviders: {
    github: {
      clientId: process.env.GITHUB_ID!,
      clientSecret: process.env.GITHUB_SECRET!,
```

## Phase 7: RBAC Migration

### 7.1 Integrate with Better Auth Admin Plugin

**Decision:** Better Auth's admin plugin provides role-based access, but might not match the complex permission system (23 permissions across 8 roles).

**Options:**
1. **Use Better Auth's role system** - Simpler, but may lose fine-grained permissions
2. **Keep custom permissions** - Store in user metadata, check manually
3. **Hybrid approach** - Use Better Auth roles + custom permissions matrix

**Recommendation:** Option 3 - Hybrid approach
- Keep `member_role` in user metadata
- Keep permission checking logic in `src/lib/auth/permissions.ts`
- Map Better Auth's session to include role/permissions

### 7.2 Update permission checks

```typescript
// In server components
const session = await auth.api.getSession({ headers: await headers() })
const role = session?.user.role ?? "mitglied"
const permissions = session?.user.permissions ?? []

if (!hasPermission(role, permissions, PERMISSIONS.ADMIN_USERS)) {
  redirect("/dashboard")
}
```

## Phase 8: Two-Factor Authentication Migration

### 8.1 Enable Better Auth 2FA Plugin

Replace custom 2FA implementation with Better Auth's built-in plugin:

```typescript
import { twoFactor } from "better-auth/plugins"

export const auth = betterAuth({
  plugins: [
    twoFactor({
      otpOptions: {
        window: 1,
      },
    }),
  ],
})
```

### 8.2 Migrate existing 2FA data

Users with `twoFactorEnabled: true` need to re-setup 2FA after migration (Better Auth uses different secret format).

**Migration strategy:**
1. Disable 2FA for all users during migration
2. Add notification to re-enable 2FA after login
3. Optionally provide manual migration for admins only

## Phase 11: Account Lockout & Rate Limiting

### 11.1 Migration decision

**Better Auth's built-in protection:**
- Basic rate limiting on auth endpoints

**Current system:**
- Custom Redis-based rate limiting
- Account lockout after failed attempts
- IP-based and email-based limits

**Recommendation:** Keep custom implementation
- Better Auth's built-in protection is basic
- Chess club app needs robust protection (DDoS prevention)
- Migrate Redis-based logic to work with Better Auth

### 11.2 Update `src/lib/auth/rate-limit.ts`

Wrap Better Auth endpoints with custom rate limiting:

```typescript
export async function authWithRateLimit(request: NextRequest) {
  await checkRateLimitByIP(ip, "login")
  return auth.handler(request)
}
```

## Risk Assessment

| Risk | Likelihood | Impact | Mitigation |
|------|-----------|--------|------------|
| Password hash incompatibility | Medium | High | Test bcrypt compatibility |
| Session data loss | Low | High | Backup before migration |
| RBAC breakage | Medium | High | Thorough testing phase |
| 2FA reset required | High | Medium | Communicate to users |

## Post-Migration Benefits

1. **Simpler architecture** - Single auth library vs NextAuth + custom code
2. **Built-in features** - 2FA, RBAC, organizations as plugins
3. **Better performance** - No separate auth server
4. **Type safety** - Better Auth is fully typed
5. **Lichess OAuth** - Native support for custom OAuth
6. **No vendor lock-in** - MIT license, self-hosted
7. **DSGVO compliance** - Data stays on Hetzner

## Decision Summary

**Proceed with migration?** YES

**Rationale:**
- Better Auth aligns with the tech stack (TypeScript, Drizzle, Next.js)
- Reduces complexity by replacing custom implementations with plugins
- Enables future features (Lichess login, organizations)
- Better long-term maintainability
- Cost: Free, open-source
- Risk: Medium, manageable with proper testing

**Key success factors:**
1. Complete database backup before migration
2. Phased rollout with thorough testing
3. Clear rollback plan
4. User communication about 2FA re-setup
